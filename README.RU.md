# HoRus-Infrastructure (RU)

<p align="center">
  <img src="../assets/logo.png" alt="HoRus Logo" width="180"/>
</p>

Платформа уровня Production для развертывания инфраструктуры в виде кода (Infrastructure-as-Code), предназначенная для подготовки систем на базе Debian к высокодоступному развертыванию сервисов.

## Обзор проекта

HoRus-Infrastructure отвечает за первичную подготовку чистых серверов Debian 13 (Trixie). Данный проект включает в себя следующие этапы:
- **Этап 1**: Начальная подготовка операционной системы и харденинг безопасности (`common` роль).
- **Этап 2**: Развертывание готового к эксплуатации (production-ready) сервера баз данных PostgreSQL 18 (`postgresql` роль).

## Структура проекта

```text
HoRus-Infrastructure/
├── ansible.cfg            # Дефолтные настройки Ansible
├── requirements.yml      # Внешние зависимости (коллекции)
├── inventory/            # Инвентарь и переменные хостов
│   ├── syslab_hosts.yml  # Единый источник истины (SoT) для хостов
│   └── group_vars/       # Групповые и секретные переменные
│       ├── all.yml        # Глобальные параметры
│       └── postgresql.yml # Параметры сервера PostgreSQL
├── bootstrap/            # Ассеты первичной подготовки
│   ├── ssh/              # Скрипты и ключи SSH
│   └── certs/            # Заготовки для PKI
├── playbooks/            # Основные точки входа (плейбуки)
│   ├── common.yml        # Точка входа для Этапа 1 (ОС bootstrap)
│   └── postgresql.yml    # Точка входа для Этапа 2 (PostgreSQL)
└── roles/                # Модульная логика инфраструктуры
    ├── common/           # Роль для базовой подготовки и защиты ОС
    ├── download/         # Абстракция для загрузки файлов через прокси
    ├── molecule_prepare/ # Подготовка среды тестирования
    ├── postgresql/       # Роль для развертывания PostgreSQL 18
    └── tests/            # Набор валидационных тестов инфраструктуры
```

## Быстрый старт

1. **Установка коллекций**:
   ```bash
   ansible-galaxy collection install -r requirements.yml
   ```

2. **Генерация паролей и секретов для инициализации**:
   Перед запуском плейбуков вам необходимо подготовить секретные переменные и SSH-ключи. Вы можете запустить этот процесс вручную или позволить плейбуку `common` сделать это автоматически.
   
   *   **Ручной запуск**:
       ```bash
       python3 bootstrap/auto_prepare.py
       ```
   *   **Что делает этот скрипт**:
       - Создает рабочий файл `inventory/group_vars/vault_inventory.yml` на основе шаблона `inventory/group_vars/vault_inventory.yml.template`, если он еще не существует.
       - Генерирует криптографически стойкие 256-символьные пароли в открытом виде и соответствующие им хеши SHA-512 crypt для административных и сервисных учетных записей.
       - Сохраняет версии паролей в открытом виде в файл `bootstrap/secrets_plain.txt` (этот файл добавлен в `.gitignore` и защищен от утечек в репозиторий). Пароли не выводятся в `stdout` терминала во избежание их утечки в логи сборщиков (CI/CD) или историю терминала.
       - Автоматически создает необходимые пары SSH-ключей ED25519 для пользователей `root`, `admin` и `jenkins` в директории `bootstrap/ssh/`.

3. **Развертывание и харденинг инфраструктуры**:
   - **Stage 1 (Подготовка и харденинг ОС)**:
     ```bash
     ansible-playbook -i inventory/syslab_hosts.yml playbooks/common.yml
     ```
   - **Stage 2 & 3.1 (Развертывание PostgreSQL и подготовка сервисных баз)**:
     ```bash
     ansible-playbook -i inventory/syslab_hosts.yml playbooks/postgresql.yml
     ```

   **Кастомизация**: По умолчанию в файле `vault_inventory.yml` (который исключен из коммитов Git) администратором указан пользователь `root`. Вы можете легко переопределить это имя (например, на `abbenden-srv`), изменить пути к приватным и публичным SSH-ключам, или задать свои пароли. Скрипт автоматической подготовки при запуске динамически считает ваши настройки из `vault_inventory.yml` и сгенерирует необходимые ключи и пароли по указанным вами путям!

## Stage 3: Подготовка баз данных, ограничение доступа и интеграция с HashiCorp Vault

Этот этап обеспечивает полное развертывание бэкендов хранения для платформенных сервисов, управление учетными записями, настройку базовой безопасности и управление секретами уровня Enterprise с помощью HashiCorp Vault.

### Stage 3.1: Подготовка сервисных баз данных и пользователей PostgreSQL
Развертывание баз данных и специализированных пользователей для платформенных сервисов с реализацией строгой политики безопасности согласно **принципу наименьших привилегий (Least Privilege Principle)**.
- **Единый источник истины (Single Source of Truth)**: Все конфигурации сервисов хранятся в общем файле `inventory/group_vars/all.yml` в структуре `platform_services`.
- **Ограничение прав доступа**: Отзываются публичные права на подключение (`PUBLIC CONNECT`) ко всем созданным базам данных, права выдаются строго владельцу БД.
- **Проверки и валидация (QA Verification)**: Проверяется корректность владельца БД и возможность TCP-подключения по SCRAM-sha-256 через локальный IP `127.0.0.1`.

### Stage 3.2 - 3.5: Оркестрация HashiCorp Vault уровня Enterprise
Развертывание надежного, безопасного и готового к эксплуатации (production-ready) кластера **HashiCorp Vault**.

1. **Высокодоступный бэкенд хранения (HA Storage)**: Настроен на работу с СУБД PostgreSQL с системной оркестрацией systemd и полным харденингом конфигурации.
2. **Автоматическая инициализация и распечатывание (Unsealing)**: При первом запуске Vault инициализируется, а ключи распечатывания и root-токен безопасно сохраняются локально в `runtime/vault/` (с правами доступа `0600` и исключением из Git). При повторных запусках ключи загружаются автоматически для плавного unseal-процесса.
3. **Механизм аутентификации AppRole (Запрет Root-токена)**: Вводится полный запрет на использование Root-токена в повседневной работе. Сервисы (`jenkins`, `gitea`, `harbor`, `authentik`) аутентифицируются по выделенным AppRole с минимальными и строго ограниченными HCL-политиками.
4. **Начальная заливка секретов (Secrets Bootstrapping)**: Использование структурированных конвейеров JSON для записи базовых паролей, TLS-ключей, Docker и GitHub доступов, что исключает утечку спецсимволов при экранировании в CLI.
5. **Продвинутая валидация SRE и мониторинг здоровья**:
   - **Сквозной Smoke-тест API**: Реальное выполнение операций записи, чтения, уничтожения версий и очистки метаданных по тестовым путям (`secret/system/healthcheck`).
   - **Smoke-тест авторизации AppRole**: Выполнение тестового входа от имени AppRole с последующей проверкой чтения секретов по выданному токену.
   - **Контекстная проверка PostgreSQL**: Подтверждение владельца таблиц и проверка прав чтения (`SELECT COUNT(*)`) под системной ролью `vault`.
   - **Проверка TLS-сертификатов**: Проверка SAN DNS/IP и валидация срока действия сертификата (>7 дней) через утилиту OpenSSL.
   - **Валидация резервных копий**: Контроль целостности и структуры локальных файлов учетных данных (`init.json`, `recovery.json`, `approles.json`).
   - **Контроль соответствия версий (Drift Check)**: Сравнение целевой и реально установленной версии Vault с логированием предупреждений.
   - **Тесты производительности (Perf Benchmark)**: Последовательные замеры задержки (latency) при работе с KV в миллисекундах (суммарное и среднее время операции).
6. **Самовосстановление после сбоев (Disaster Recovery)**: Автоматическое устранение дрейфа конфигурации (config drift), восстановление владельцев таблиц и прав в СУБД, восстановление удаленных/измененных файлов конфигурации `vault.hcl` и юнитов systemd.

## Полезные команды

- **Проверка связи (Ping) со всеми хостами**: `ansible all -i inventory/syslab_hosts.yml -m ping`
- **Проверочный режим (Dry-run)**: `ansible-playbook -i inventory/syslab_hosts.yml playbooks/common.yml --check`
- **Запуск только на конкретном хосте (Limit)**: `ansible-playbook -i inventory/syslab_hosts.yml playbooks/common.yml --limit horus-db-srv01`
- **Запуск всех плейбуков базовой подготовки**:
  ```bash
  ansible-playbook -i inventory/syslab_hosts.yml playbooks/common.yml
  ansible-playbook -i inventory/syslab_hosts.yml playbooks/postgresql.yml
  ```
- **Запуск плейбука PostgreSQL**:
  ```bash
  ansible-playbook -i inventory/syslab_hosts.yml playbooks/postgresql.yml
  ```


## Контроль качества и тестирование (QA)

Для поддержания высокого качества кода и надежности инфраструктуры в проект интегрированы инструменты статического анализа и автоматического тестирования:

*   **yamllint**: Инструмент для проверки синтаксиса и форматирования файлов YAML (пробелы, отступы, переносы строк и валидность структур). Конфигурация описана в `.yamllint`.
    *   *Запуск вручную*: `yamllint .`
*   **ansible-lint**: Специализированный анализатор для проверки плейбуков, ролей и конфигураций Ansible на соответствие лучшим практикам и внутренним стандартам. Конфигурация описана в `.ansible-lint`.
    *   *Запуск вручную*: `ansible-lint`
*   **pre-commit**: Инструмент автоматического запуска проверок перед фиксацией изменений (коммитом). Работает **исключительно локально на рабочей машине разработчика** еще до того, как коммит создается в Git. Он полностью независим от серверных платформ хостинга Git (таких как Gitea, GitHub или GitLab).
    *   *Установка хуков*: `pre-commit install`
    *   *Принудительный запуск всех проверок перед коммитом*: `pre-commit run --all-files`
*   **Molecule**: Фреймворк для изолированного тестирования ролей Ansible. Molecule запускает тестовый сценарий, применяет роль и проверяет успешность компиляции и выполнения всех задач.
    *   *Драйвер*: В качестве драйвера контейнеров используется **Podman** для обеспечения полной изоляции сред тестирования.
    *   *Целевая ОС*: Тестирование выполняется непосредственно в контейнере с **Debian 13 (Trixie)**, что гарантирует точное соответствие продуктовой целевой среде.
    *   *Роль подготовки*: Использует роль `molecule_prepare` для автоматической установки предварительных зависимостей (Python 3, python3-apt, sudo) в чистый контейнер перед тестированием основной роли `common`.
    *   *Запуск тестов*: `molecule test`

## Архитектура и долгосрочное развитие

### 1. Единый источник истины (HoRus-Inventory)
В настоящее время инвентарный файл хостов находится в `inventory/syslab_hosts.yml`. Однако перспективная архитектура предполагает выделение инвентаря в отдельный независимый репозиторий **`HoRus-Inventory`**, который станет единственным **Источником истины (Source of Truth, SoT)** для всех Ansible-проектов экосистемы.
*   **Source of Truth (SoT)**: Служит единым и неделимым реестром всех метаданных хостов, групп и переменных для всех репозиториев HoRus.
*   Это полностью исключает дублирование списков хостов и переменных, предотвращая расхождение конфигураций (configuration drift).

### 2. Модульный пайплайн и зоны ответственности
Каждый репозиторий в семействе HoRus имеет строго одну область ответственности. Они абсолютно независимы и взаимодействуют последовательно:

```
HoRus-Bootstrap
        │
        ▼ (Создает физические или виртуальные серверы, т.е. VM/LXC)
        │
HoRus-Inventory (Source of Truth)
        │
        ▼ (Регистрирует метаданные хоста и его переменные)
        │
HoRus-Infrastructure (Bootstrap Infrastructure / OS Hardening)
        │
        ▼ (Выполняет базовую подготовку ОС, ставит пакеты, настраивает SSH)
        │
HoRus-Control-Plane
        ├── Мониторинг (Prometheus/Grafana)
        ├── Логирование и Observability (Grafana Loki, OpenTelemetry)
        ├── Алёртинг
        └── Общие сервисы ядра (Authentik IAM, Wiki, Gitea)
```

### 3. Автоматический запуск (Bootstrap) при появлении нового хоста (Future Work)
Проект разработан с учетом поддержки событийного запуска (event-driven execution):
*   **Событие**: После регистрации нового хоста в центральном источнике истины **Source of Truth** (`HoRus-Inventory`) внешняя система оркестрации (например, Jenkins, AWX, Semaphore UI или любой другой планировщик) выполняет запуск плейбука:
    ```bash
    ansible-playbook playbooks/common.yml --limit <host>
    ```
*   **Разделение ответственности (Decoupled Orchestration)**: Этот репозиторий **намеренно не реализует логику самой оркестрации или планирования**. Его цель — предоставить чистый, модульный и полностью идемпотентный механизм начальной настройки серверов. Система запуска полностью отделена от содержимого репозитория.
*   **Результат**: Новый хост автоматически подготавливается, на нем настраивается безопасность, создаются пользователи и обновляются пакеты без необходимости ручного вмешательства администратора.
