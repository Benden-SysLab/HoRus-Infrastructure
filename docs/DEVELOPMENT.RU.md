# Процесс разработки инфраструктуры и жизненный цикл качества (QA)

Данный документ описывает полный процесс разработки (workflow) и стандарты обеспечения качества (QA) для проекта `HoRus-Infrastructure`. Соблюдение этого жизненного цикла гарантирует, что любые вносимые изменения будут тщательно проверены, отформатированы и протестированы до их отправки в общий репозиторий.

---

## Шаги разработки (Step-by-Step)

Каждый цикл внедрения новых функций, изменения ролей или обновления плейбуков состоит из следующих последовательных шагов:

```
[1] Изменение кода (роли/плейбука)
          │
          ▼
[2] Проверочный запуск (ansible-playbook --check)
          │
          ▼
[3] Статический анализ YAML (yamllint)
          │
          ▼
[4] Анализ лучших практик Ansible (ansible-lint)
          │
          ▼
[5] Локальные интеграционные тесты (molecule test)
          │
          ▼
[6] Локальный Git коммит (git commit)
          │
          ▼
[7] Автоматический pre-commit хук (выполняется локально)
          │
          ▼
[8] Пуш в удаленный репозиторий (git push)
```

---

### Шаг 1: Изменение кода

Внесите необходимые изменения в файлы соответствующей роли (например, `roles/common/`) или плейбуков (например, в каталоге `playbooks/`).
*   Если вводятся новые переменные, обязательно определите их дефолтные значения или обновите шаблоны.
*   Соблюдайте принцип обратной совместимости.

### Шаг 2: Проверочный запуск (Dry-run)

Перед реальным запуском проверьте, компилируются ли ваши плейбуки и проходят ли они базовую проверку на целевых хостах разработки:
```bash
ansible-playbook -i inventory/syslab_hosts.yml playbooks/common.yml --check
```

### Шаг 3: Статический анализ YAML (`yamllint`)

Проверьте, что все измененные YAML-файлы строго соответствуют правилам форматирования проекта (отступы, пробелы, переносы строк):
```bash
yamllint .
```
*Конфигурация описана в файле `.yamllint`.*

### Шаг 4: Анализ лучших практик Ansible (`ansible-lint`)

Запустите `ansible-lint` для поиска устаревшего синтаксиса, потенциальных ошибок и несоответствий лучшим практикам Ansible:
```bash
ansible-lint
```
*Конфигурация описана в файле `.ansible-lint`.*

### Шаг 5: Локальное интеграционное тестирование через Molecule (`molecule test`)

Протестируйте изменения локально на целевой операционной системе — **Debian 13 (Trixie)** — в полностью изолированном контейнере с помощью Podman:
```bash
molecule test
```
*   Molecule автоматически запустит легковесный контейнер Podman на базе официального образа `debian:13-slim` без тяжелого и проблемного systemd cgroup окружения.
*   Он выполнит вспомогательную роль `molecule_prepare` для автоматической установки минимальных зависимостей (Python 3, python3-apt, sudo) прямо в чистый контейнер.
*   Затем соберет факты Ansible и применит основную роль `common`, проверяя корректность ее компиляции и успешность выполнения.

### Шаг 6: Создание локального Git коммита

Добавьте проверенные файлы в индекс и запустите создание коммита:
```bash
git add .
git commit -m "feat: implement secure user provisioning and SSH hardening"
```

### Шаг 7: Автоматическая проверка хуками (`pre-commit`)

При выполнении команды `git commit` фреймворк **`pre-commit`** перехватывает коммит и автоматически запускает цепочку проверок на измененных файлах:
1.  `trailing-whitespace` — удаляет лишние пробелы в концах строк.
2.  `end-of-file-fixer` — гарантирует наличие пустой строки в конце файлов.
3.  `check-merge-conflict` — предотвращает случайное добавление маркеров конфликтов слияния Git.
4.  `check-added-large-files` — блокирует добавление файлов крупнее 500 КБ.
5.  `check-yaml` — проверяет синтаксис YAML-структур.
6.  `yamllint` — запускает статический анализ YAML.
7.  `ansible-lint` — проверяет измененные роли и плейбуки Ansible.

> 💡 **Важно:** `pre-commit` работает **исключительно локально на вашей рабочей станции**. Он не имеет абсолютно никаких зависимостей от серверных Git-платформ (таких как Gitea, GitHub или GitLab). Это персональный щит разработчика от отправки некачественного кода еще до того, как изменения покинут локальную машину.

*   **Совет:** Принудительно запустить все хуки pre-commit на всех файлах в любое время можно командой:
    ```bash
    pre-commit run --all-files
    ```

### Шаг 8: Пуш изменений

После того как локальные хуки завершились успешно и коммит был зафиксирован в локальной истории, отправьте изменения в удаленный репозиторий:
```bash
git push origin main
```

## Преимущества данного подхода

Использование этого жизненного цикла гарантирует:
1.  **Безупречное качество**: Исключается попадание неформатированного кода, синтаксических ошибок или неработающих плейбуков в удаленный репозиторий.
2.  **Защита коллег от багов**: Команда может затягивать изменения, будучи на 100% уверенной в их стабильности.
3.  **Демонстрация экспертизы**: Данный структурированный процесс разработки демонстрирует высокий профессионализм, культуру написания кода и понимание DevOps-практик, что критически важно для сильного портфолио перед потенциальными заказчиками или работодателями.
