# Ansible Coding Standards

To ensure code readability, maintainability, and standardisation across the whole engineering team, all Ansible playbooks, roles, and tasks must strictly follow these coding standards.

---

## 1. Modules Over Shell / Command Commands

Always prefer dedicated, built-in Ansible modules over using raw command line execution tools (`shell` or `command`).
*   **Anti-pattern (DO NOT USE):**
    ```yaml
    - name: Update apt cache
      command: apt-get update
    ```
*   **Best Practice (USE):**
    ```yaml
    - name: Update APT cache
      ansible.builtin.apt:
        update_cache: true
    ```
*   *Exception:* Only use `shell` or `raw` commands if a specialized module does not exist for the task (e.g. bootstrap step inside clean container environments), and always declare `changed_when: false` or implement proper checks.

---

## 2. Fully Qualified Collection Names (FQCN)

Always use Fully Qualified Collection Names (FQCN) for all tasks. This prevents resolution conflicts and ensures predictable executions across differing Ansible controller versions.
*   **Bad:**
    ```yaml
    - apt:
        name: htop
    ```
*   **Good:**
    ```yaml
    - ansible.builtin.apt:
        name: htop
    ```

---

## 3. Explicit Declarations of State & Booleans

Never rely on implicit defaults for actions or variables. Write clear, unambiguous, explicit keys:
*   Always specify `state` parameter (e.g. `state: present`, `state: started`).
*   Always use native YAML booleans: `true` and `false` (in lowercase, without quotes). Avoid legacy aliases like `yes`, `no`, `on`, `off`.

---

## 4. Sensitive Variables via Defaults

Never hardcode magical variables, passwords, or direct settings inside task files. 
*   **Defaults**: Declare default configuration variables inside the role's `defaults/main.yml`.
*   **Vars**: Store private, non-overrideable operational constants inside `vars/`.
*   **Vaults / Secrets**: All private keys, passwords, and sensitive information must be parsed through the Ansible Vault or handled dynamically via secure password processors (such as `.gitignore` secrets files generated by pre-flight preparation engines).

---

## 5. Descriptive Task Names

Every single task must contain a clear, professional, and human-readable descriptive `name` string. 
*   **Bad:**
    ```yaml
    - ansible.builtin.user:
        name: admin
    ```
*   **Good:**
    ```yaml
    - name: Ensure administrative user exists with sudo permissions
      ansible.builtin.user:
        name: admin
    ```

---

## 6. Handler Integration with Notify

Whenever a task induces a configuration state change that requires restarting a daemon or refreshing a caching layer, never execute a restart step inside inline tasks. Use the `notify` directive to trigger the appropriate system handler.
*   **Best Practice:**
    ```yaml
    - name: Template SSH configuration file
      ansible.builtin.template:
        src: sshd_config.j2
        dest: /etc/ssh/sshd_config
      notify: Restart SSH
    ```

---

## 7. Restricted Use of Ignore Errors

The parameter `ignore_errors: true` is strictly prohibited unless there is an absolute, well-documented technical justification (e.g. checking if a tool exists that may throw a non-zero exit code). Always prefer to use `failed_when` or catch failures via `register` and evaluate blocks.

---

## 8. Strictly Clean Style Indentations

*   Use exactly `2` spaces for indentation level.
*   Never use tab characters.
*   Separate tasks, playbooks, and structural elements using clear spacing.
*   Strictly check your styling conventions using `yamllint` and `.yamllint` standard configurations before making commits.
