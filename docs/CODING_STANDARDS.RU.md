# Стандарты разработки Ansible (Coding Standards)

Чтобы обеспечить читаемость кода, простоту его сопровождения и стандартизацию в рамках всей инженерной команды, все плейбуки, роли и задачи Ansible должны строго соответствовать следующим стандартам разработки.

---

## 1. Использование специализированных модулей вместо shell / command

Всегда отдавайте приоритет встроенным и специализированным модулям Ansible перед запуском утилит командной строки через `shell` или `command`.
*   **Антипаттерн (НЕ ИСПОЛЬЗОВАТЬ):**
    ```yaml
    - name: Update apt cache
      command: apt-get update
    ```
*   **Лучшая практика (ИСПОЛЬЗОВАТЬ):**
    ```yaml
    - name: Update APT cache
      ansible.builtin.apt:
        update_cache: true
    ```
*   *Исключение:* Использование `shell` или `raw` допускается только в тех редких случаях, когда для задачи не существует специализированного модуля (например, первичный bootstrap в чистых контейнерах). При этом обязательно объявляйте параметр `changed_when: false` или обрабатывайте статус изменения.

---

## 2. Использование полных путей модулей (FQCN)

Всегда указывайте Fully Qualified Collection Names (FQCN) для всех вызываемых модулей. Это предотвращает конфликты версий и гарантирует предсказуемое выполнение на любых контроллерах Ansible.
*   **Плохо:**
    ```yaml
    - apt:
        name: htop
    ```
*   **Хорошо:**
    ```yaml
    - ansible.builtin.apt:
        name: htop
    ```

---

## 3. Явное описание состояний и булевых значений

Никогда не полагайтесь на неявные значения по умолчанию. Пишите прозрачный, недвусмысленный код:
*   Всегда явно указывайте параметр `state` (например, `state: present`, `state: started`).
*   Всегда используйте нативные YAML-булевы значения: `true` и `false` (строчными буквами, без кавычек). Не используйте устаревшие синонимы `yes`, `no`, `on`, `off`.

---

## 4. Вынесение переменных в Defaults

Никогда не встраивайте (hardcode) константы, пароли или специфические настройки непосредственно в файлы задач (tasks).
*   **Defaults**: Объявляйте дефолтные конфигурационные параметры в каталоге роли `defaults/main.yml`.
*   **Vars**: Храните закрытые, непереопределяемые внутренние константы в каталоге `vars/`.
*   **Vaults / Secrets**: Все чувствительные данные (ключи, пароли) должны обрабатываться через Ansible Vault либо динамически генерироваться нашими скриптами подготовки в локальные файлы конфигурации, добавленные в `.gitignore` (например, `secrets_plain.txt`).

---

## 5. Понятные и описательные имена задач (name)

Каждая задача Ansible должна содержать понятное, профессиональное и информативное поле `name`.
*   **Плохо:**
    ```yaml
    - ansible.builtin.user:
        name: admin
    ```
*   **Хорошо:**
    ```yaml
    - name: Ensure administrative user exists with sudo permissions
      ansible.builtin.user:
        name: admin
    ```

---

## 6. Использование обработчиков (handlers) и notify

Если выполнение задачи изменяет состояние системы, требующее перезапуска службы или сброса кэша, никогда не делайте перезапуск прямо в блоке задач. Используйте директиву `notify` для вызова соответствующего обработчика (handler).
*   **Лучшая практика:**
    ```yaml
    - name: Template SSH configuration file
      ansible.builtin.template:
        src: sshd_config.j2
        dest: /etc/ssh/sshd_config
      notify: Restart SSH
    ```

---

## 7. Ограничение использования ignore_errors

Параметр `ignore_errors: true` строго запрещен, кроме редких случаев с полным документированием технической необходимости. Вместо этого используйте `failed_when` или сохраняйте результат выполнения через `register` с дальнейшим анализом в блоке `block/rescue`.

---

## 8. Форматирование кода и отступы

*   Используйте ровно `2` пробела для каждого уровня отступов.
*   Использование символов табуляции строго запрещено.
*   Разделяйте блоки задач, плейбуки и структуры визуальными пробелами.
*   Перед выполнением коммитов обязательно проверяйте стилистическую валидность ваших файлов с помощью `yamllint`.
