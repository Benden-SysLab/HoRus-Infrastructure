# Архитектурные принципы

Этот документ определяет ключевые архитектурные правила и принципы проектирования экосистемы `HoRus`. Соблюдение этих правил гарантирует высокую сопровождаемость, тестируемость, отказоустойчивость и чистоту кодовой базы нашей инфраструктуры.

---

## Общая архитектура экосистемы

Чтобы исключить жесткую связанность кода инфраструктуры и инвентарей, экосистема HoRus разделена на специализированные репозитории с четкими зонами ответственности:

```
                  ┌──────────────────────┐
                  │   HoRus-Bootstrap    │
                  └──────────┬───────────┘
                             │
                             ▼ (Создает физ./вирт. хосты: VM/LXC)
                  ┌──────────────────────┐
                  │   HoRus-Inventory    │◀─── [Источник истины (SoT)]
                  └──────────┬───────────┘
                             │
                             ▼ (Регистрирует хосты, группы и переменные)
                  ┌──────────────────────┐
                  │ HoRus-Infrastructure │◀─── [Текущий проект]
                  └──────────┬───────────┘
                             │
                             ▼ (Базовая настройка ОС, пакетов и hardening)
                  ┌──────────────────────┐
                  │  HoRus-Control-Plane │
                  └──────────┬───────────┘
                             │
                             ├───────► Мониторинг (Prometheus, Grafana)
                             ├───────► Логирование (Grafana Loki, OpenTelemetry)
                             ├───────► Алёртинг и уведомления
                             └───────► Базовые сервисы (Authentik IAM, Wiki, Gitea)
```

---

## Архитектурные принципы

### 1. Одна роль — одна зона ответственности
Каждая Ansible-роль решает строго одну задачу (например, базовая настройка ОС, управление пакетами, подготовка среды тестирования). Роль никогда не должна пытаться и разворачивать виртуальные серверы, и одновременно настраивать сложное прикладное ПО.

### 2. Один плейбук — одно действие
Каждый плейбук координирует выполнение конкретного высокоуровневого сценария (например, первичная подготовка серверов через `playbooks/common.yml`). Избегайте создания гигантских монолитных плейбуков общего назначения.

### 3. Инфраструктура идемпотентна
Запуск плейбука один раз, два раза или сто раз должен приводить к абсолютно одинаковому конечному состоянию целевой системы. Любая задача должна быть безопасной при повторных запусках. Если задача по природе не идемпотентна, используйте условные операторы и параметры `changed_when` для поддержания стабильности.

### 4. Инфраструктура не привязана к средствам оркестрации
Наши роли и плейбуки полностью независимы от систем запуска (Jenkins, AWX, Semaphore UI и др.). Репозиторий предоставляет чистый и надежный исполнительный механизм, а внешние системы оркестрации выступают только в качестве планировщиков.

### 5. Инвентарь — единый источник истины (Source of Truth, SoT)
Все метаданные, группы и параметры хостов должны быть вынесены в централизованный репозиторий `HoRus-Inventory`. Отдельные проекты не дублируют списки хостов во избежание расхождения конфигураций (configuration drift).

### 6. Секреты никогда не хранятся в Git
Пароли, приватные ключи SSH и TLS-сертификаты строго запрещено фиксировать в истории контроля версий. Скрипты генерации (`auto_prepare.py`) должны безопасно записывать локальные, добавленные в `.gitignore` секреты и резервные plaintext-пароли (`secrets_plain.txt`).

### 7. Роли тестируются изолированно
Каждая роль должна проверяться отдельно. В данном проекте используется фреймворк **Molecule** с драйвером контейнеров **Podman** на базе **Debian 13 (Trixie)** для оперативного и воспроизводимого тестирования всех задач.

### 8. Контроль качества (QA) обязателен
Любое изменение перед слиянием должно успешно проходить:
-   Статический анализ YAML (`yamllint`)
-   Проверку на соответствие лучшим практикам Ansible (`ansible-lint`)
-   Конвергентные тесты в контейнере (`molecule test`)
-   Локальный запуск Git-хуков перед коммитом (`pre-commit`)
