# Инженерные стандарты и руководство по разработке HoRus

Этот документ определяет единые инженерные принципы, стандарты качества и лучшие практики для всех репозиториев экосистемы **HoRus**. Соблюдение этих стандартов обязательно для обеспечения поддерживаемости, безопасности, масштабируемости и единообразия инфраструктуры на всех этапах её жизненного цикла.

---

## 🏛️ Ключевые инженерные принципы

### 1. Одна роль — одна ответственность (Single Responsibility)
Каждая роль Ansible, скрипт или компонент должны выполнять **ровно одну задачу** и делать это безупречно.
- Избегайте монолитных ролей, настраивающих несколько разнородных сервисов (например, не совмещайте установку СУБД и настройку веб-сервера).
- Держите плейбуки лаконичными, делегируя настройку конкретных компонентов модульным и изолированным ролям.

### 2. Максимальное использование встроенных модулей (Native First)
Всегда отдавайте приоритет встроенным и проверенным модулям Ansible (`ansible.builtin.*`, `community.general.*`) вместо запуска прямых Bash-команд через `ansible.builtin.shell` или `ansible.builtin.command`.
- **Почему**: Нативные модули Ansible из коробки обеспечивают отслеживание изменений (state tracking), корректную обработку ошибок и кроссплатформенность.
- **Исключение**: Использование `shell` / `command` допускается только тогда, когда специализированного модуля не существует, либо для инициализации кастомных бинарных файлов. При этом вы **обязаны** использовать параметр `changed_when` для корректного отслеживания изменений.

### 3. Обязательная идемпотентность (Strict Idempotency)
Любой плейбук, роль, задача и вспомогательный скрипт должны быть полностью **идемпотентными**.
- Запуск плейбука один, два или сто раз подряд на одной и той же системе должен приводить к одному и тому же предсказуемому результату, не вызывая нежелательных побочных эффектов или повторных лишних действий.
- Все кастомные скрипты автоматизации (такие как `generate_ssh_keys.sh` или скрипт предварительной подготовки `auto_prepare.py`) должны сначала проверить текущее состояние системы и вносить изменения только в случае реальной необходимости.

### 4. Никаких секретов в системе контроля версий (Zero Secrets in Git)
Категорически запрещено коммитить в Git пароли в открытом виде, приватные SSH-ключи, API-токены или реальные хеши паролей пользователей.
- Для локальной разработки и тестирования используйте файл `vault_inventory.yml` (и шаблон `.template`), который надежно заблокирован в `.gitignore`.
- Для промышленной эксплуатации (production) используйте Ansible Vault, переменные окружения или внешние системы управления секретами (HashiCorp Vault, облачные Secret Managers).
- Исключайте из коммитов любые потенциально уязвимые файлы с самого первого коммита.

### 5. Все значения через переменные (No Magic Constants)
Использование «магических констант» и жестко захардкоженных настроек строго запрещено.
- Выносите имена пользователей, порты, пути к ключам, адреса прокси-серверов и списки пакетов в `group_vars`, `host_vars` или в значения по умолчанию (`defaults`) внутри ролей.
- Держите структурные и часто меняющиеся параметры логически сгруппированными в YAML-структурах.

### 6. Четкие границы жизненного цикла (Lifecycle Boundaries)
Каждый репозиторий экосистемы HoRus отвечает строго за свой этап подготовки и эксплуатации инфраструктуры и не должен брать на себя задачи смежных этапов.
- **Stage 1 (HoRus-Infrastructure)**: Занимается исключительно базовой подготовкой ОС (обновление, пакеты, часовые пояса, системные пользователи, безопасность SSH). Он **не должен** разворачивать среды выполнения контейнеров, веб-серверы или прикладные базы данных.
- **Stage 2 (HoRus-Control-Plane)**: Разворачивает оркестраторы, сетевые фабрики и системные сервисы (Docker, Kubernetes, пулы СУБД).
- **Stage 3 (HoRus-Applications)**: Разворачивает прикладные медиа- и ИИ-сервисы поверх подготовленного слоя управления.

## 🛠️ Стандарты написания Ansible

### Именование задач (Task Naming)
Каждая задача обязана содержать понятное человеку поле `name`, описывающее бизнес-логику выполняемого действия на английском языке (или русском, если согласовано проектом), а не технический синтаксис:
```yaml
# ❌ Плохо
- name: run apt
  ansible.builtin.apt:
    name: nginx

#  Отлично
- name: Install Nginx Web Server Package
  ansible.builtin.apt:
    name: nginx
    state: present
```

### Форматирование
- Соблюдайте стандартный синтаксис YAML: **отступы ровно в 2 пробела**.
- Всегда вызывайте модули по их полному имени (например, `ansible.builtin.template` вместо `template`).
- Шаблоны Jinja2 (`.j2`) должны содержать комментарии, указывающие на структуру переменных.

## 🤝 Как внести свой вклад

1. **Инициализация окружения**: Запустите скрипт pre-flight проверок `python3 bootstrap/auto_prepare.py` для автоматической подготовки локальных ключей и секретов.
2. **Ветки разработки**: Создавайте ветки от `main` с именами, отражающими этап и фичу (например, `feature/stage1-chrony-hardening`).
3. **Проверка**: Всегда выполняйте синтаксический анализ и запуск плейбука в режиме тестирования (`ansible-playbook --check`) перед отправкой изменений.
4. **Pull Requests**: Следите за тем, чтобы изменения в PR строго соответствовали границам зоны ответственности текущего репозитория.
